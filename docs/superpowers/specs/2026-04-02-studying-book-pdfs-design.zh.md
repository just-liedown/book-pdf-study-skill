# 书籍 PDF 学习 Skill 设计

**目标：** 创建一个可复用的 Codex skill，用于围绕“书籍型 PDF”开展学习，并且避免每轮都重新读取整份 PDF。

## 问题定义

书籍 PDF 往往很长，反复全量读取既慢又浪费上下文。这个 skill 的核心是先做一次预处理，把可复用的学习材料持久化下来，然后再围绕学习目标、当前水平和学习进度来驱动后续交互。

## 第一版范围

第一版包含：

- 判断一个 PDF 是否更像书籍/教材，而不是普通文档
- 对 PDF 做一次预处理，生成可复用产物
- 先确认用户想学到什么程度，以及当前掌握程度
- 支持三种显式模式：`/learn`、`/ask`、`/test`
- 持久化学习状态，避免上下文丢失

第一版暂不包含：

- 对纯扫描版 PDF 的 OCR
- 脱离当前学习会话的长期复习排程
- 外部数据库或复杂 UI

## 用户模型

这个 skill 需要先确认两件事：

- 学习目标：入门、只学指定章节、系统理解、面试准备、项目应用、深度精读
- 当前水平：零基础、略懂、学过但忘了、已经读过一部分

之后需要持续维护：

- 当前处于哪个学习阶段
- 章节或主题进度
- 掌握度与薄弱点
- 最近的 `/test` 结果

## 核心流程

1. 检查 PDF，判断它是否像一本书。
2. 首次运行时做预处理，并把缓存写到源 PDF 对应的缓存目录。
3. 对整本书做结构、难度、知识密度和前置要求的初步评估。
4. 询问用户的目标深度与当前水平。
5. 进入命令驱动模式：
   - `/learn` 推进当前学习阶段
   - `/ask` 解释某个难点、概念、章节或原文片段
   - `/test` 做检测并更新进度

## 学习阶段

这些学习范式不应该做成互斥按钮，而应该做成阶段化流转。

1. 定位阶段：确认目标、水平、范围
2. 建图阶段：梳理全书结构和依赖关系
3. 理解阶段：按章节学习并给出通俗解释
4. 深化阶段：通过追问暴露理解漏洞
5. 应用阶段：用例题、案例、面试题做迁移
6. 巩固阶段：整理成高频复习材料
7. 检测阶段：仅在用户显式输入 `/test` 时进入

推荐范式：

- 建图阶段：提纲挈领
- 理解阶段：章节精读 + 费曼讲解
- 深化阶段：苏格拉底追问
- 应用阶段：例题驱动 + 面试问答
- 巩固阶段：闪卡复习

## 三种模式

### `/learn`

按当前阶段推进学习，输出要依据学习目标和当前掌握度调整，而不是机械地按页总结。

### `/ask`

用于处理局部卡点，支持两种入口：

- 精确输入：用户贴原文、页码、段落
- 模糊输入：用户只说章节、概念、主题

回答时至少覆盖：

- 这段内容在说什么
- 你可能卡在哪里
- 作者为什么会这么写
- 它和上下文、全书主线的关系
- 更直白的改写，以及必要时的例子

### `/test`

只在用户显式触发时进入。可以针对当前阶段、指定章节或目标层级来测。测完以后要更新掌握度，并建议继续推进、局部补强，或者回退补学。

## 持久化产物

预处理后应生成一个稳定的缓存目录。第一版建议包含：

- `manifest.json`：源文件路径、哈希、页数、生成时间
- `pages.jsonl`：按页抽取后的文本
- `chunks.jsonl`：带页码关联的检索块
- `toc_candidates.txt`：可能是目录或标题的文本行
- `book-analysis.md`：对结构、难度、前置知识的分析
- `study-state.json`：目标、水平、阶段、进度、测试历史

这样后续对 `/learn`、`/ask`、`/test` 的处理就可以直接依赖缓存，而不用每轮重新读取整本 PDF。

## 实现策略

第一版预处理脚本基于本机已有的 `pdftotext`，避免依赖额外 Python 包。脚本负责抽取文本、保留页边界、做基本清洗，并输出固定目录结构。

Skill 主文档需要指导 Codex：

- 当缓存缺失或过期时先运行预处理
- 根据不同模式，只读取必要的缓存文件
- 在 `/learn` 和 `/test` 后更新 `study-state.json`
- 在 `/ask` 中尽量只读取相关页或相关 chunk

## 验证

第一版至少需要完成：

- 用 `quick_validate.py` 校验 skill 结构
- 对预处理脚本做一次冒烟测试
- 人工检查 skill 描述是否能在“学习书籍 PDF”的请求上正确触发
